거래 영역

-주문 (Order)

주문 스냅샷 생성 (사용자 정보 + 상품 정보 저장)
주문 상태 관리 및 이벤트 발행
비회원 주문번호 + 비밀번호 조회

-결제 (Payment)

모의 결제 (포인트, 현금, 카드)
결제 실패 / 중복 결제 / 부분 환불 케이스 검증 및 처리

-재고 (Inventory)

결제 시 재고 차감 (-1) 및 취소 시 복구
재고 관리 레이스 컨디션(동시성) 해결
재고 증감 로그 기록

-클레임 (Claim)

취소 / 환불 / 교환 요청 수집
요청 > 승인/반려 > 처리 완료 상태 관리
상품 수거 지시

전시 및 마케팅 영역

-상품 (Product)

카테고리, 키워드, 가격 범위, 정렬(최신/인기/가격순) 관리
상품 등록 / 판매 중단 / 삭제 / 수정
상품 페이지(상세설명)
가격 정책 (정가/판매가/할인가)

-프로모션 (Promotion)

쿠폰/포인트 발급 (랜덤박스, 관리자, 이벤트) 및 복구
발급 및 사용 제한 (카테고리, 기간, 1회성, 중복사용 여부, 회원 등급)

-장바구니 (Cart)

장바구니 담기, 수정, 삭제

회원 및 지원 영역

-회원 (Member)

소셜 로그인 + 일반 회원가입
개인정보, 주소록 관리
Wishlist 관리 (품절/판매중단 시 표시 처리, 카테고리/가격 변동 반영)
회원 이력(Event/History) 기록

-배송 (Delivery)

배송 상태 변경 (ㅇㅇ허브 입고 > ㅇㅇ허브 출고 > 집 도착)
시간 차이에 따른 자동 상태 변경 (x시간 초과 시)
부분 배송 처리, 배송지 변경 가능 시점 관리

-리뷰 (Review)

구매자 검증 및 리뷰 작성 (사진, 별점, 시간)
리뷰 수정/삭제 정책, 신고/블라인드 처리

-고객 센터 (CS)

상담 챗봇
상담 티켓 관리 및 답변

공통 및 인프라 영역

-알림 (Notification)

이메일 발송 전담
배송 상태 알림, 상담 답변 알림 메일 전송
템플릿 관리 및 메일 전송 로그 기록

-백오피스 (Backoffice)

프론트엔드 연동: 별도 제작된 관리자 페이지에 데이터 제공 (API Gateway 역할)
계정 관리: 관리자 계정 및 권한 관리
통합 로그 (Audit): 각 도메인에서 발생하는 이벤트(상품 변경, 주문 취소 처리 등)를 구독하여 누가, 언제, 무엇을 조작했는지 통합 이력 저장

프론트엔드:
	프레임워크: Next.js (App Router)
	배포/호스팅: Netlify

백엔드:
	언어: java 
	프레임워크: spring boot

서비스 디스커버리(Eureka) 도입으로 프로젝트끼리 소통하도록

데이터베이스 및 인프라
	인메모리 DB: Redis
	메인 DB: H2

Docker:Redis 실행

DB는 스키마를 논리적 격리, 객체 참조(@ManyToOne)를 금지, ID 기반 참조(memberId)만 사용

모든 도메인을 프로젝트로 분리하는건 여건상 어려운 관계로 관계도 별로 도메인을 묶은 후,
모듈러 모노리스 형태로 만듦, 그것들을 MSA처럼 연결하는 하이브리드 방식으로 진행 예정

설계방식은 레이어드 아키텍쳐 구조를 따를 예정

3대 Windows PC 분산 배치 계획 (Draft)

PC 1번

인프라: Redis (이 PC에만 Docker를 깔아서 빠르고 깔끔하게 1개 띄움)

API 게이트웨이: 외부 요청을 받아 PC 2, PC 3으로 라우팅해주는 문지기 역할 (Spring Cloud Gateway)

[모듈 1] Core Commerce (트랜잭션/돈과 직결된 영역)

주문 (Order): 전체 흐름의 중심
결제 (Payment): 주문과 동시에 이루어져야 함
재고 (Inventory): 결제 성공 시 무조건 깎여야 함
프로모션 (Promotion): 결제 시 쿠폰/포인트가 즉시 차감되어야 함
배송 (Delivery): 결제 완료 시 바로 배송 준비 상태로 넘어가야 함
클레임 (Claim): 취소/환불 시 위 모든 것(재고, 결제, 쿠폰)이 한 번에 롤백되어야 함

이유: 트래픽과 데이터 정합성이 가장 중요한 핵심 서비스와 중앙 저장소(Redis)를 묶어 네트워크 지연을 최소화합니다.

PC 2번

[모듈 2] Catalog & Engagement (탐색 및 사용자 상호작용 영역)

상품 (Product): 전시, 검색, 카테고리
회원 (Member): 개인정보, 로그인, 위시리스트
리뷰 (Review): 상품에 종속되는 사용자 피드백
장바구니 (Cart): 회원과 상품을 이어주는 휘발성 데이터 (PC 1번의 Redis 활용)

이유: 조회(Read)가 가장 많이 일어나는 도메인들을 모아 별도의 PC 자원을 풀로 활용하게 합니다.

PC 3번

프론트엔드: Next.js 웹 서버 구동 (쇼핑몰 화면 및 백오피스 화면)

[모듈 3] Operation & Support (운영 및 비동기 지원 영역)

백오피스 (Backoffice): 관리자 활동 이력 및 통합 권한 관리
알림 (Notification): 메일 전송 등 외부 시스템 연동 (지연이 발생해도 메인 서버에 영향 없음)
고객 센터 (CS): 상담 티켓, 챗봇

이유: 프론트엔드 렌더링에 집중하면서, 메인 결제 로직
